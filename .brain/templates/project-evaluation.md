# Template: Project Evaluation

> Lightweight decision framework for evaluating whether to pursue, revive, or kill a project. Adapted from the 1-pager concept — stripped down for a solo builder, not a cross-functional team.

## Frontmatter

```yaml
---
id: DOC-xxx
created: YYYY-MM-DD
updated: YYYY-MM-DD
type: knowledge
domain: [relevant domain]
audience: personal
status: active
---
```

## Structure

### What is it?
[2-3 sentences. What does this project do? Who is it for?]

### Why now?
[What has changed that makes this worth pursuing/reviving? New technology? Market shift? Personal bandwidth? A signal from intake?]

### What does it cost?
- **Time:** [hours/week estimate]
- **Money:** [hosting, tools, services]
- **Opportunity cost:** [what doesn't get done if this gets attention]

### What's the upside?
- **Revenue potential:** [realistic near-term, optimistic long-term]
- **Learning value:** [skills built, patterns learned]
- **Portfolio value:** [does this showcase something valuable for career/brand?]
- **Connection to $15k/month goal:** [how does this move the escape velocity needle?]

### Profile Alignment
[Read profile.md — how does this connect to Andrew's active projects, interests, and goals? Does it compound with anything already in motion, or is it a standalone bet?]

### Dependencies
[What must be true for this to work? Access to an API? A certain number of users? Time from the day job?]

### Go / No-Go / Shelf
- [ ] **Go** — pursue actively, add to roadmap
- [ ] **No-Go** — kill it, capture learnings, move on
- [ ] **Shelf** — not now, but worth revisiting when [condition]

### Decision Notes
[Why this decision? What would change the answer?]
